PR & 코드 리뷰 가이드

팀원의 승인 없이는 코드가 반영되지 않습니다.
안전한 협업을 위한 PR 프로세스를 알아보세요.

Pull Request · Code Review · Approve · Merge — AIBizTeam9

왜 바로 반영이 안 되는 걸까?

우리 저장소의 main 브랜치는 보호(protected)되어 있습니다. 아무나 직접 코드를 올리면 실수로 서비스가 망가질 수 있기 때문입니다.

그래서 모든 코드는 최소 1명의 팀원이 확인(Approve)한 뒤에만 main에 반영됩니다.

이 과정을 PR(Pull Request) → Code Review → Approve → Merge라고 합니다.

💡 비유로 이해하기

📝 팀 보고서에 비유하면...

main 브랜치 = 교수님께 제출하는 최종 보고서

내 브랜치 = 내가 작업하는 초안 문서

PR(Pull Request) = "내 초안을 최종 보고서에 넣어주세요" 라는 요청서

Code Review = 팀원이 초안을 읽어보고 검토하는 것

Approve = "괜찮아 보여, 넣어도 돼" 라는 승인

Merge = 승인 받은 초안을 최종 보고서에 합치는 것

🍪 더 쉬운 비유

카페에서 새 메뉴를 추가한다고 상상해보세요.

1. 직원(나)이 새 레시피를 만듦 → 내 브랜치에서 코딩
2. "이 메뉴 추가해도 될까요?" 요청 → PR 생성
3. 매니저(팀원)가 레시피를 맛보고 검토 → Code Review
4. "OK, 메뉴판에 올려" 승인 → Approve
5. 메뉴판에 정식 등록 → Merge

우리 저장소의 규칙

📜 브랜치 보호 규칙

우리 저장소 main 브랜치에는 다음 규칙이 적용되어 있습니다:

🚫 규칙 1: main에 직접 push 금지

main 브랜치에 직접 코드를 올릴 수 없습니다. 반드시 본인 브랜치에서 작업하고 PR을 통해 반영해야 합니다.

✅ 규칙 2: 최소 1명의 승인(Approve) 필수

PR을 올리면, 다른 팀원 최소 1명이 코드를 확인하고 "Approve"를 눌러야 main에 합칠(Merge) 수 있습니다.

  • 본인이 올린 PR은 본인이 승인할 수 없습니다.
  • 승인 없이는 Merge 버튼이 비활성화됩니다.
왜 이런 규칙이 있을까?
  • 실수로 버그가 있는 코드가 main에 들어가는 것을 방지
  • 팀원 모두가 코드 변경 사항을 파악할 수 있음
  • 실무에서도 동일한 방식으로 협업하므로, 미리 익히는 연습

👥 두 가지 역할

PR 과정에서 팀원은 두 가지 역할을 번갈아 하게 됩니다:

📝 PR 작성자
내가 만든 코드를 올리는 사람
"이거 확인해주세요!"
🔍 리뷰어 (Reviewer)
팀원의 코드를 확인하는 사람
"확인했어, 좋아 보여!"
모든 팀원이 두 역할 모두 해야 합니다. 내 코드를 올리기도 하고, 다른 팀원의 코드를 검토해주기도 합니다.
역할 A — PR 작성자 (코드를 올리는 사람)
1

🚀 PR 만드는 법

본인 브랜치에서 작업을 완료하고 push한 뒤, GitHub에서 PR을 생성합니다.

방법 A: GitHub에서 뜨는 알림 이용 (가장 쉬움)

  1. push 후 GitHub 저장소 페이지에 접속합니다.
  2. 상단에 노란색 배너가 뜹니다: "본인브랜치 had recent pushes — Compare & pull request"
  3. "Compare & pull request" 초록색 버튼을 클릭합니다.

방법 B: 직접 PR 생성

  1. 저장소 페이지에서 "Pull requests" 탭을 클릭합니다.
  2. "New pull request" 초록색 버튼을 클릭합니다.
  3. base: main ← compare: 본인브랜치 를 선택합니다.
  4. "Create pull request"를 클릭합니다.

PR 작성 시 입력할 내용

작성 항목
Title (제목): 무엇을 했는지 한 줄 요약
  예: "퀴즈 페이지 UI 추가"

Description (설명): 자세한 변경 내용
  예: "5개 질문을 순서대로 보여주는 퀴즈 페이지를 만들었습니다"

Reviewers (검토자): 우측 사이드바에서 팀원을 선택
  → 리뷰해줄 팀원을 지정합니다
PR 생성 완료! 이제 팀원이 리뷰하고 Approve해줄 때까지 기다립니다.
카톡이나 슬랙으로 "PR 올렸어, 리뷰 부탁해!" 라고 알려주면 더 빨라집니다.
2

💬 리뷰 피드백 반영하기

팀원이 코드에 수정 요청(Request Changes)을 남길 수도 있습니다. 이때 당황하지 마세요!

  1. PR 페이지에서 팀원이 남긴 코멘트를 확인합니다.
  2. VS Code로 돌아가서 해당 부분을 수정합니다.
  3. 수정한 코드를 다시 commit & push합니다.
  4. PR에 수정한 내용이 자동으로 반영됩니다. (새로 PR을 만들 필요 없음!)
같은 브랜치에 push하면 PR이 자동 업데이트됩니다. PR을 새로 만들지 마세요!
역할 B — 리뷰어 (코드를 확인하는 사람)
3

🔍 코드 리뷰하는 법

팀원이 PR을 올리면, 변경된 코드를 확인하고 의견을 남깁니다.

리뷰 시작하기

  1. 저장소의 Pull requests 탭에 접속합니다.
  2. 리뷰할 PR을 클릭합니다.
  3. 상단의 "Files changed" 탭을 클릭합니다.
  4. 변경된 코드를 확인합니다:
    • 초록색 줄 = 새로 추가된 코드
    • 빨간색 줄 = 삭제된 코드
코딩을 몰라도 리뷰할 수 있어요! 바이브 코딩 프로젝트에서는 코드 문법보다 결과물이 제대로 동작하는지가 더 중요합니다. "이 기능이 잘 되는 것 같다" 또는 "이 부분 글씨가 이상하다" 같은 피드백도 훌륭한 리뷰입니다.

리뷰할 때 확인할 것

  • 어떤 파일이 추가/수정되었는지 대략 확인
  • 한국어 UI 텍스트에 오타가 없는지
  • PR 설명에 적힌 기능이 코드에 반영된 것 같은지
  • 이상해 보이는 부분이 없는지 (파일이 너무 많이 삭제되었다거나)
4

✅ 승인(Approve) 또는 수정 요청하기

코드를 확인한 후 판정을 내립니다.

승인/수정 요청 방법

  1. "Files changed" 탭에서 코드를 확인한 뒤
  2. 우측 상단의 "Review changes" 초록색 버튼을 클릭합니다.
  3. 세 가지 옵션 중 하나를 선택합니다:
💬 Comment
의견만 남기기
(승인/거절 아님)
✅ Approve
승인! 코드 반영 OK
(이걸 눌러야 Merge 가능)
🔄 Request Changes
수정 필요
(고친 뒤 다시 리뷰)
  1. 코멘트 입력창에 한 마디 남깁니다. (예: "확인했어, LGTM!", "잘 만들었다!")
  2. "Submit review" 버튼을 클릭합니다.
LGTM이란? "Looks Good To Me"의 약자로, "나한테 좋아 보인다 = 승인" 이라는 뜻의 개발자 용어입니다.
주의: "Comment"는 승인이 아닙니다! 반드시 "Approve"를 선택해야 Merge가 가능해집니다.
최종 단계 — Merge (코드 반영)
5

🎉 Merge하기

1명 이상의 Approve를 받으면 드디어 main에 코드를 합칠 수 있습니다!

  1. PR 페이지 하단으로 스크롤합니다.
  2. 초록색 "Merge pull request" 버튼이 활성화된 것을 확인합니다.
  3. 버튼을 클릭합니다.
  4. "Confirm merge"를 한 번 더 클릭합니다.
Merge 완료! 본인의 코드가 main에 성공적으로 반영되었습니다!
PR 상태가 Merged로 바뀌면 성공입니다.
Merge 버튼이 회색이고 안 눌린다면?
→ 아직 팀원의 Approve를 받지 못한 것입니다. 팀원에게 리뷰를 요청하세요!
실전 시나리오

🎬 시나리오: 동근이 퀴즈 페이지를 만들었다

1. 동근: 본인 브랜치에서 코딩 & push
dongkeun 브랜치에서 퀴즈 페이지를 만들고 git push
2. 동근: GitHub에서 PR 생성
제목: "퀴즈 페이지 추가" / Reviewer로 지윤을 지정
3. 동근 → 지윤: "PR 올렸어, 리뷰 부탁해!"
카톡으로 알림
4. 지윤: PR 확인 & 코드 리뷰
Files changed 탭에서 변경된 내용 확인
5. 지윤: Approve 승인!
Review changes → Approve → "LGTM!" → Submit review
6. 동근: Merge!
Merge pull request 클릭 → main에 코드 반영 완료!

🎬 시나리오: 수정 요청을 받았다

1. 재림: PR 올림
결과 페이지 디자인 변경 PR을 생성
2. 석빈: 리뷰 후 수정 요청
Request Changes → "버튼 색상이 배경과 너무 비슷해서 안 보여!"
3. 재림: 코드 수정 후 다시 push
버튼 색상 수정 → commit → push (같은 브랜치에!)
4. 석빈: 다시 확인 후 Approve
"이제 잘 보여, LGTM!"
5. 재림: Merge!
수정된 코드가 main에 반영 완료
수정 요청은 나쁜 게 아닙니다! 더 좋은 코드를 만들기 위한 과정이에요. 편하게 의견을 주고받으세요.

❔ FAQ

Q: PR을 올린 뒤에 코드를 더 수정해도 되나요?

네! 같은 브랜치에 계속 push하면 PR에 자동으로 반영됩니다. 새 PR을 만들 필요 없어요.

Q: 누가 Merge 버튼을 누르나요?

보통 PR을 올린 본인이 Approve를 받은 후 Merge합니다. 팀 내에서 다르게 정해도 괜찮습니다.

Q: Approve 없이 Merge할 수 있나요?

아니요. 우리 저장소는 최소 1명의 Approve가 없으면 Merge 버튼이 비활성화됩니다.

Q: 내가 올린 PR을 내가 Approve할 수 있나요?

아니요. 본인이 올린 PR은 본인이 승인할 수 없습니다. 반드시 다른 팀원이 해야 합니다.

Q: 리뷰를 빨리 받고 싶어요!

PR을 올린 후 카톡/슬랙으로 팀원에게 직접 알려주세요. GitHub 알림은 놓치기 쉽습니다.

Q: Merge 후에 내 브랜치는 어떻게 되나요?

Merge 후 GitHub에서 "Delete branch" 버튼이 나타납니다. 눌러서 정리해도 되고, 그대로 두고 계속 사용해도 됩니다.

계속 사용할 경우, 다음 작업 전에 main의 최신 코드를 가져오세요:

git checkout main
git pull
git checkout 본인브랜치
git merge main

📚 한 줄 요약

내 브랜치에서 코딩 → Push → PR 생성 → 팀원 Approve → Merge

이게 전부입니다! 팀원의 승인 한 번이면 코드가 반영됩니다.
